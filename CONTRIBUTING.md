# Contributing to Awesome Privacy Tools

Thanks for helping improve **Awesome Privacy Tools**. The goal is to maintain a trustworthy, practical list of privacy tools.

## What Belongs Here

A submission should usually meet most of these requirements:

- It is a tool, app, service, protocol, library, operating system, self-hosted project, or practical privacy utility.
- It clearly improves privacy, security, anonymity, encryption, data minimization, anti-tracking, or user control.
- It is useful to a real audience: beginners, power users, developers, journalists, activists, teams, or self-hosters.
- It has a clear website, repository, documentation page, or public explanation.
- It is maintained, historically important, or still technically relevant.
- It avoids deceptive privacy claims, surveillance advertising, dark patterns, unnecessary telemetry, or unclear ownership.
- It has a reasonable trust model. Open source is preferred, but not mandatory.

## What Does Not Belong Here

Please avoid submitting:

- Affiliate links, referral links, coupon-only pages, paid placements, or SEO landing pages.
- Thin content, copied listicles, or projects with unclear privacy value.
- Tools primarily intended for abuse, harassment, credential theft, surveillance, or evading accountability for harm.
- Unmaintained projects with unresolved security issues unless they are clearly marked as historically important.
- Duplicate entries or near-duplicate services that do not add a meaningful new option.
- Marketing copy copied directly from a vendor site.
- Cryptocurrency, gambling, adult, or weapon-related tools unless there is a clear and mainstream privacy/security reason to include them.

## Entry Format

Use this format:

```markdown
- [Name](https://example.com/) - One clear sentence explaining what it does and why it is useful.
```

Style rules:

- Keep descriptions concise, factual, and neutral.
- End descriptions with a period.
- Use HTTPS links whenever possible.
- Link to the official website first. If the website is weak or unavailable, link to the repository.
- Do not use tracking parameters such as `utm_source`, `ref`, `fbclid`, or affiliate codes.
- Add new entries alphabetically within a section unless a section intentionally groups beginner resources first.
- Avoid hype such as “best,” “ultimate,” “military-grade,” or “100% anonymous” unless independently verifiable.

## Pull Request Checklist

Before opening a pull request, confirm:

- [ ] I searched the README to avoid duplicates.
- [ ] I added the entry to the most relevant section.
- [ ] I used the required Markdown format.
- [ ] The link works and does not contain tracking parameters.
- [ ] The description is original and not copied from marketing copy.
- [ ] I disclosed any affiliation with the project.
- [ ] I understand that maintainers may edit wording for clarity, neutrality...

## Suggesting a New Category

New categories are welcome when they make the list easier to navigate. A category should usually have at least three high-quality entries or a strong reason to exist.

Good category examples:

- Browser Privacy Extensions
- Private Search Engines
- Self-Hosted Privacy Tools
- Password Managers, Passkeys, and Authentication

## Removing or Updating Entries

Please open a pull request or issue if a resource is:

- Dead or redirected to unrelated content.
- No longer maintained and risky to recommend.
- Acquired or changed in a way that affects privacy claims.
- Replaced by a better-maintained project.
- Misleadingly described.

When removing a resource, include a short reason so reviewers can understand the decision.

## Project-Affiliated Submissions

Maintainers, employees, founders, investors, contractors, affiliates, and community managers may submit their own projects. Please disclose the affiliation in the pull request.

Good disclosure examples:

- “Disclosure: I maintain this project.”
- “Disclosure: I work for this company.”
- “Disclosure: I am a user and occasional contributor.”

Undisclosed promotion may be closed without review.

## Review Philosophy

This is a curated list, not a complete directory. A declined submission does not mean a project is bad. It may simply be too niche, too new, too similar to existing entries, or outside the current scope.

Maintainers may prioritize:

- Clear privacy benefit.
- Usability.
- Security model transparency.
- Longevity and maintenance.
- Community trust.
- Diversity of options across platforms and threat models.

## Reporting Security or Abuse Concerns

If an entry appears malicious, compromised, abandoned with serious vulnerabilities, or unsafe for users, open an issue with supporting details. Avoid posting exploit instructions or sensitive private information in public issues.
